{/*
  起草大纲（按 cue 顺序）：

  two-clocks     — 视觉走 rAF（屏幕刷新率），声音走 AudioContext
                   （硬件采样钟）。两只钟会缓慢漂移。我们要保证的
                   不是"它们绝对同步"——是"它们在用户感知的时间
                   尺度上不出戏"。
  scheduling     — Web Audio 的调度模式：100ms tick + 250ms lookahead，
                   每次往前调度 [now, now+250ms] 之间还没安排的音符。
                   游标缓存避免每次重扫。
  anchors        — 把音符 onset ms → 屏幕 x。anchors 是 [t, x] 序列。
                   xAtMs 走二分 + 线性插值。msAtX 反向。
  retiming       — 一个微妙问题：MEI 的 `<tempo>` 标记 tstamp 在小节
                   起点，但视觉上字形（"Allegretto"）在小节中段才
                   出现。如果按 tstamp 触发 overlay 切换，文字会
                   在还没到达播放头时就换。解：算字形 x 位置，反
                   推它"过播放头"那一刻的 ms。
  tail-and-loop  — 非 loop 谱：曲子结束后留 1.8s 让最后一颗音视觉
                   上走出舞台。loop 谱：用相位算 + 整数代数把
                   translate 在 (-musicWidth, 0] 之间无缝 wrap。
  drag           — 用户拖动谱面要把视觉位置反推回 ms（msAtX），
                   这就是为什么需要双向插值。
  recap          — 收：两只钟，永远在做对齐。用户看到的"准"是
                   一种被持续修补的错觉。
*/}
