{/*
  起草大纲（按 cue 顺序）—— 全流程组件思路。
  这篇是「形状」的故事，不是「难题」的故事；难题留给《看错的尺》。

  empty      — 起。把谱子嵌进文章不是新事，但「能跟着读、能听一段、能拖
               一格」的谱子是。一句话引出 MEI / MusicXML / Verovio。这篇讲
               的不是怎么把音符画出来——讲的是怎么把它扎成一个组件。

  strip      — 整套架构的地基：`breaks: 'none'` + 巨大 pageWidth +
               adjustPageHeight。一句 setOption 决定后面的一切：曲子
               摊成一条横向的带子。如果分页了，所有后续逻辑都不一样。

  pipeline   — MEI → Verovio → SVG → DOM 注入。每一段的职责。
               静态的（曲子结构、装饰、调号变化）一次性烧进 SVG；
               动态的（播放头位置、tempo overlay 切换）由运行时处理。
               所以渲染发生一次，播放发生千次。

  layers     — 舞台里的几层叠在一起，互不干扰：
                 pan         — Verovio 出的 SVG，每帧靠 translate3d 平移
                 staff-lines — 单独画 5 条横线，单一来源避免双笔加粗
                 chrome      — 「冻结的报头」(clef/keysig/meter)，假装不动
                 playhead    — 一根固定的竖线
                 hint        — "click to play"
               这是为什么 chrome 能假装不动而 pan 在动——它们物理上分层。

  lifecycle  — 不一上来就挂。IntersectionObserver 等到滚到视口附近才
               让 Verovio 醒来；离开页面（astro:before-swap）拆得干净。
               WASM 是大件，缓存（in-memory + IndexedDB）让重访免重渲。

  ticking    — 演奏时两只钟一起转：
                 视觉钟 (rAF)        ——  从音频钟读 ms → 锚点查 x →
                                        translate3d 设位置
                 音频钟 (AudioContext) —— 100ms tick + 250ms lookahead
                                        独立排音符，一个游标记到哪了
               两只钟在同一首曲子上跳同一支舞，谁也不主导。

  recap      — 整套的形：一条带子 + 几层平行的 view + 一只眼盯着时间。
               这是写给「一个人想看自己谱子」的，不是「任何人想用一个
               通用乐谱播放器」的——所以它的边界设得很死，但里面的每
               一处都是为这一个用例选的。
*/}
